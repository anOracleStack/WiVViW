☽ What the Schema Missed ☽
Ten experiential & operational systems from Clotho's vision
have no database home in Gemini's architecture.
These are not optional. They are load-bearing.
1 — Worthiness Scoring
The onboarding intake generates an internal
seriousness & alignment score per applicant.
Gemini's wivviw_users has a status field but no
structured scoring model or progression audit trail.

Required table: worthiness_scores
user_id, dimension, score, evaluator_notes, created_at
2 — Auto-Advance Permissions
Three execution modes per engine, per user:
full_auto · single_approval · manual_precision

No table exists for storing per-engine permission states.

Required table: advance_permissions
user_id, engine_key, mode, updated_at
3 — Quality Scoring Engine
Per-task evaluation with evolving category frameworks.
The "above-threshold" logic means WiVViW's standards
must self-upgrade when new dimensions of excellence appear.

Required tables:
scoring_templates — category sets per task type
scoring_results — per-task, per-category scores
scoring_evolution — audit trail of framework upgrades
4 — Outbound Approval Queue
Nothing leaves WiVViW ungoverned.
Emails, posts, brand statements, assets — all gated.

Required table: outbound_queue
id, user_id, content_type, payload, status,
optimization_notes, approved_at, sent_at
5 — Connected Stack Registry
Discover, connect, observe, & eventually replace
external tools the user depends on.

Required tables:
connected_apps — user_id, app_name, connection_status, config
stack_usage_log — app_id, action, frequency, replacement_candidate
6 — Budget Envelopes
Token/cost estimation before execution.
Hard caps, refill alerts, BYOK key storage.

Required table: budget_envelopes
user_id, hard_cap, current_spend, alert_threshold,
byok_keys (encrypted), model_preferences
7 — Engine Health Snapshots
Context Pressure · Verification Coverage · Conflict Rate
Stability · Cost Burn · Extraction Fidelity · Continuity Risk

Required table: engine_health_snapshots
session_id, engine_key, metrics_json, captured_at
8 — Moral Universe States
Accepted Cosmos → Limbo / Review → Inferno / Banished
Future system. Not Phase 1. But the schema must anticipate it.

Required: Add standing_state & standing_history
to wivviw_users. standing_state ENUM: accepted, limbo, banished.
standing_history as JSONB array of state transitions.
9 — MOIRAI Package Typing
Gemini's moirai_packages captures output.
But Clotho defined three distinct package types:
Deliverable · Audit · Thought

Required: Add package_type ENUM to moirai_packages
Values: deliverable, audit, thought
Add: score NUMERIC, pass_threshold NUMERIC, passed BOOLEAN
10 — Proactive Optimization Queue
The anti-back-and-forth engine's structured inference:
what was asked, what should happen next, what would improve it.

Required table: optimization_queue
session_id, proposed_actions JSONB, status,
user_decision (auto/approved/rejected), executed_at
· · · · ·
With these ten additions, every experiential system
Clotho envisioned has a persistent, queryable home.
The schema is now load-bearing across all layers.

Part V — Critical Blind Spots Caught
What Neither Clotho Nor Lachesis Addressed
───────────────────────────────────────────────────────────────
1 — Rate Limiting & Abuse Protection
A selective platform that allows heavy multi-model runs
is a target for token-drain abuse. Even accepted users
can accidentally or intentionally burn through resources.

Required: Per-user rate limiting at the API route level.
Upstash Redis or Supabase-native rate limiting.
Enforce before Inngest dispatch, not after.
2 — Secrets Management for BYOK
Storing user API keys (BYOK) in budget_envelopes
as encrypted JSONB is not sufficient long-term.

Required: Supabase Vault or an external secrets manager.
Phase 1 can use encrypted columns with pgcrypto.
Phase 2 should migrate to Vault.
3 — Idempotency in MOIRAI Runs
Inngest handles retries. But if a MOIRAI step partially
completes, rerunning it must not create duplicate packages.

Required: Idempotency keys on moirai_packages.
Composite unique: (session_id, step_number, package_type)
4 — Provider Fallback Logic
If OpenAI is down, the simple pipeline halts at Step 1.
If Gemini rate-limits, Step 2 stalls indefinitely.

Required: Provider health check + fallback routing.
If primary provider fails: retry 2x → fallback to
alternate provider → log degradation in engine_health_snapshots.
5 — Session Continuity Across Browser Loss
Odyssey sessions can span hours. If the user closes
the browser mid-run, the Inngest job continues server-side.
But the UI has no way to reconnect to a running job.

Required: Polling endpoint + Supabase Realtime subscription
on odyssey_sessions for live status updates.
User reopens → immediately sees current state.
