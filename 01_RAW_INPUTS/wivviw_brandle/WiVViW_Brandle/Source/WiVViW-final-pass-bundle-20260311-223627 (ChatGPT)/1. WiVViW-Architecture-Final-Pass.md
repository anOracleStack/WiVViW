# WiVViW — Definitive Architecture, Build Wedge & Phase 1 Blueprint
## Final Synthesis — METHOD MOIRAI Stage IV

---

## PART I — RECONCILIATION

### What Eunoia Got Right
Eunoia correctly preserved the load-bearing architecture already established for WiVViW:

- WiVViW is the master operating system, not a generic AI wrapper
- dRANb is the naming/identity decision engine
- drandL is the digital foothold engine
- 4IELD is field / market / competitor intelligence
- 4TRESS is protection / safeguard guidance
- the Truth Kernel is the first real moat
- MOIRAI is the model-orchestration law
- quality scoring is mandatory, repeatable, and evolving
- nothing important should leave the system ungoverned
- selective access and intake are not decoration; they are part of the product philosophy

Eunoia also correctly identified that Gemini's earlier engineering schema missed multiple systems that were already present in the experiential and operating vision.

### What Must Be Tightened Beyond Eunoia
Eunoia's architecture is strong, but two things needed to be corrected for reality:

**1. Phase 1 was still too heavy for the user's actual timeline.**  
The architecture is valid, but the execution order needed a **Phase 0 / Survival Wedge** before the heavier private-beta hardening plan. WiVViW needs a sharp, credible proof artifact fast, then the deeper infrastructure.

**2. "Thought packages" must not depend on raw hidden chain-of-thought capture.**  
WiVViW should store **structured reasoning notes**, not rely on extracting internal hidden reasoning from providers.  
Canonical package types therefore become:
- **Deliverable Package**
- **Audit Package**
- **Reasoning Notes Package**

This preserves auditability without depending on raw chain-of-thought access.

### Final Rulings on Eunoia's Additions
Below is the final ruling on the major additions introduced in the Eunoia pass.

| Addition | Keep? | Phase | Final ruling |
|---|---:|---:|---|
| Prompt Registry | Yes | Phase 1 | Essential. Without versioning, quality regressions are invisible. |
| Session Forking | Yes | Phase 2 | Valuable, but not needed for first proof wedge. Keep in schema/docs, defer build. |
| Context Compression Layer | Yes | Phase 1 | Essential once MOIRAI runs exceed safe context budgets. |
| Event Bus / Realtime Updates | Yes | Phase 1.5 | High value, but can be deferred until after first working end-to-end wedge. |
| Export Pipeline | Yes | Phase 1 | Essential for credibility and user trust. |
| Rollback Path | Yes | Phase 1 | Essential. WiVViW cannot regress silently. |
| Provider Health Monitor | Yes | Phase 1 | Keep, but start simple. No need for overengineered heartbeat mesh on day one. |

### Final Additions Beyond Eunoia
These were either missing or understated and are now canon.

**A. Phase 0 / Survival Wedge**
WiVViW needs a compressed "show-don't-tell" release path:
- a presentable landing / intake surface
- a working dRANb → drandL path
- Truth Kernel states visible
- receipts generated
- exportable proof artifact
- enough polish to sell, demo, and raise with

This is not the full platform. This is the first holy artifact.

**B. Admin Audit Trail**
Any high-trust premium system needs full internal traceability:
- who approved a user
- who denied
- who changed prompt versions
- who changed scoring templates
- who overrode outputs
- who re-ran tasks

Required table: `admin_actions`

**C. Artifact Store**
Exports, generated reports, screenshots, decision packets, and approved outbound items need durable references and checksums.

Required table: `artifacts`
- id
- session_id
- artifact_type
- storage_path
- checksum
- metadata
- created_at

**D. Run Controls / Feature Flags**
Because WiVViW will evolve quickly, engines and sub-features need flags:
- enable/disable auto-advance
- turn on/off provider fallback
- hide incomplete engines
- stage experimental scoring logic

Required table: `feature_flags`

**E. Consent & Approval Log**
For outbound communications and sensitive actions:
- who approved
- what was approved
- when
- under which mode (auto / single approval / manual)

Required table: `approval_log`

**F. Test & Fixture Discipline**
The architecture cannot stop at docs + routes. Every core subsystem needs:
- fixtures
- deterministic test cases
- known-failure examples
- regression tests

This belongs in file structure and build order immediately.

### Schema Notes

- `package_type` ENUM (`deliverable`, `audit`, `reasoning_notes`)
- `score`
- `pass_threshold`
- `passed`
- `rollback_of` (nullable FK)
- unique constraint on `(session_id, step_number, package_type, attempt_number)`

- content hash / checksum for evidence
- source family

- normalized verdict payload

**`budget_envelopes`** should include:
- `hard_cap`
- `alert_threshold`
- `current_spend`
- `mode` (`managed`, `byok_prepped`)
- `provider_preferences`

## PART V — BLIND SPOTS CAUGHT

### From Eunoia and This Final Pass

1. **Rate limiting must occur before dispatch**, not after the expensive work has already started.

2. **Prompt drift is lethal** unless prompt versions are tracked and linked to output quality.

3. **Raw chain-of-thought should not be a dependency.**  
Use structured reasoning notes packages instead.

4. **No admin audit trail = no high-trust system.**

5. **No artifact checksum store = no durable credibility.**

6. **No rollback path = no safe optimization loop.**

7. **No test fixtures = regression will silently rot the platform.**

8. **No feature flags = every partial feature becomes dangerous.**

9. **No approval log = outbound governance is unverifiable.**

10. **No Phase 0 wedge = architecture exists, but nothing sells.**

11. **No continuity compression = token ceiling silence returns through the back door.**

12. **No complexity classifier = MOIRAI wastes money internally and violates its own anti-waste philosophy.**
